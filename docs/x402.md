# Troof API (x402) — design

Expose Troof's analysis + verification as **pay-per-call HTTP APIs** using the
[x402](https://x402.org) payment protocol (HTTP `402 Payment Required` + an on-chain
stablecoin micropayment), so any app or agent can embed **verifiable Sui analysis**.

## Endpoints (planned)

| Method | Path | Price* | Returns |
|---|---|---|---|
| `GET` | `/api/v1/token/score?coinType=&network=` | $0.01 | Troof Score + line items |
| `GET` | `/api/v1/wallet?address=&network=` | $0.02 | integrity-checked wallet report |
| `POST` | `/api/v1/seal` | $0.05 | seals the report → `{ blobId, proofUrl, anchor }` |
| `GET` | `/api/v1/verify/:blobId` | free | re-fetch + on-chain hash check verdict |

\* indicative; settled in USDC.

## Flow

1. Client calls an endpoint with no payment → server replies **`402`** with x402 `accepts`
   (asset, amount, pay-to address, network).
2. Client pays and retries with the `X-PAYMENT` header.
3. Server verifies the payment via an x402 **facilitator**, then runs the request and returns
   the data — plus, for paid analysis, a **sealed Troof proof** so the *API response itself is
   independently verifiable*. (A verifiable API: you can prove what the API told you.)

## Why it fits Troof

- The verify endpoint stays free (the moat should be open); the value-add analysis is metered.
- Each paid response can carry a `proofUrl`, making Troof the first **verifiable** data API for Sui.

## Status

Spec only in this repo. Implementation plan: an x402 middleware on `/api/v1/*` (e.g. the `x402-next`
package) wrapping the existing `report` / `token` / `score` / `seal` libs. Settlement starts on
USDC (Base) with Sui settlement on the roadmap.
